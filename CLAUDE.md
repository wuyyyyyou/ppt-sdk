## Repo layout

Three top-level subprojects, all glued together by JSON-RPC plugin calls:

- `ppt-app/` — the Anna App shell (static SPA + manifest + local executa shims).
  Runs inside `anna-app dev` for local debugging.
- `presenton-template-engine/` — owns template discovery, deck HTML rendering,
  HTML→PPTX model conversion, and the new `app_*` workspace API consumed by
  ppt-app.
- `presenton-pptx-generator/` — Python plugin that writes the final `.pptx`
  from the PPTX model JSON.

`scripts/run-ppt-pipeline.mjs` chains the three steps from a manifest. The
VS Code default task drives that script.

## ppt-app: the invariants that bite

The Anna app shell is intentionally thin. Workflow logic lives in
`ppt-engine` app-facing tools, not in the frontend.

- **Tool IDs must match in 5 places** (changing one without the others breaks
  local dev): `manifest.json` `required_executas[].tool_id` + `ui.host_api.tools`,
  `src/api/annaPptBackend.ts` defaults or `.env`, `executas/*/executa.json`,
  and the real plugin `describe` manifest name. Current IDs:
  - `tool-lightvoss_5433-ppt-engine-6443rj2a`
  - `tool-lightvoss_5433-ppt-gener-dc7ftcep`
- **The standalone dev server has been removed.** Local development is
  `anna-app dev` only. Don't reintroduce a Vite dev server proxy path.
- **MainStage starts at `template`**, not `brief`. User picks a template
  first; selection forks the template into `<workspace>/template/` and
  writes `template.json`.

## Template preview pipeline (load-bearing — recently rebuilt)

The picker uses **static image URLs**, not data URLs. Image bytes never go
through the JSON-RPC channel.

- Source of truth: `presenton-template-engine/dist/template-previews/groups/<group>/*.png`,
  generated by `npm run build:full` or `npm run build:template-previews`
  (Puppeteer-based, slow, gated behind explicit script — plain
  `npm run build` does NOT regenerate).
- Plain `npm run build` runs through `tsup clean`, which would wipe the
  previews. A `preserve-dist-template-previews.mjs` save/restore step
  protects them.
- Frontend consumption: `ppt-app/scripts/sync-template-previews.mjs` copies
  `dist/template-previews/groups/<group>/*.png` into
  `ppt-app/public/template-previews/<group>/`. Wired into ppt-app's
  `predev` / `prebuild` hooks so `npm run dev` and `npm run build` both
  auto-sync. `public/template-previews/` is gitignored.
- Backend `app_list_template_groups` returns each group's `preview` (cover)
  and `previews[]` (all images) as `{ url, width, height, layout_id, ... }`
  refs. URL format: `template-previews/<group>/<file>.png`, same-origin.
- Frontend renders `<img src={preview.url}>` directly. There is no
  `app_get_template_preview` call from the frontend anymore — that tool
  still exists server-side but is unused. Delete when convenient.

**If a new preview doesn't show up in the picker:**
1. Did you rebuild previews? `cd presenton-template-engine && npm run build:full`
2. Did you re-sync? `cd ppt-app && npm run sync:template-previews` (or
   re-run `npm run dev`/`build`, which trigger it).

## Anna runtime + JSON-RPC constraints

- **Anna runtime does not yet natively understand `__file_transport` pointer
  responses.** The plugin always returns `invoke` responses via file
  transport, and `ppt-app/executas/ppt-engine-local/ppt_engine_local.js`
  is a protocol adapter that reads the pointer file and inlines the JSON
  back into stdout for the runtime. **Do not delete that adapter** until
  the runtime gains native pointer support.
- The adapter has a **1 MB inline cap**. Tools returning more than that
  through file transport will get a clean JSON-RPC error instead of
  crashing the runtime. If you see "refusing to inline N bytes" in
  stderr, the tool needs to return a URL/reference rather than embedding
  the payload.
- **Architectural rule going forward**: any tool response larger than ~64 KB
  should return a URL/path/artifact ID, not the bytes themselves. Image
  data, generated decks, screenshots — all references, never inlined.

## Future: AI-modified TSX preview

When the AI agent edits TSX in `<workspace>/template/`, **do not** reuse the
static-image picker path or render those edits in ppt-app's own bundle.
That's untrusted runtime code and needs:

- A separate dev render server / iframe with error isolation
- Compile (esbuild/swc) → render → screenshot pipeline owned by the engine
- Same render path that produces the final deck (so preview ≡ final output)

The static-image picker is for the **fixed builtin** template set only.
Dynamic edits get a different surface.

## Build & validate order

When something is "stale", build in this order:

1. `cd presenton-template-engine && npm run build` (or `build:full` if
   template previews changed)
2. `cd ppt-app && npm run build` (auto-runs the preview sync first)
3. `cd ppt-app && npm run validate` to confirm manifest + bundle pass
   `anna-app validate --strict`

For UI iteration without rebuilding the engine:

```bash
cd ppt-app && npm run dev          # uses real Anna LLM bridge
cd ppt-app && npm run dev:mock-llm  # deterministic outline fixture
cd ppt-app && npm run dev -- --no-llm  # UI only, no outline tests
```

`anna-app login` is required for real-LLM mode.

## Pitfalls worth not rediscovering

- `Agent.md` lives in repo root but is in `.gitignore`. It still works for
  local Claude/agent reading, but don't expect remote collaborators to see
  edits. CLAUDE.md (this file) is the right place to push canonical agent
  guidance into the repo.
- `presenton-template-engine` ships dual ESM/CJS bundles. `import.meta.url`
  paths need ESM-safe resolution; `__dirname` paths need CJS-safe. The
  build-preview script imports the **ESM** entry explicitly
  (`dist/index.js`) to avoid the trap.
- Puppeteer runs are slow and need browser privileges. If you're sandboxed,
  `npm run build:template-previews` will fail and ask you to escalate.
  `npm run build` (without `:full`) skips Puppeteer.
- Removing `__file_transport` handling from `ppt_engine_local.js` will
  break **every** tool call, not just large ones — see runtime constraint
  above.
