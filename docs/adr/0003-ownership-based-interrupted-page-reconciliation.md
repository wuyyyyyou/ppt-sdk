# 基于归属(而非时间戳)的中断页面对账

PPT 逐页生成的运行循环只在 app 内执行,且应用视图是 `single_instance`,因此任何处于活跃状态(`authoring`、`rendering` 等)的 Page Generation Unit,只要在"打开工作区"或"停止结算后"这两个确定无主的时机被观察到,就一定不再被任何运行持有。据此引入终态 `interrupted`:在这些确定无主的时机把活跃页对账并持久化为 `interrupted`,而不是用时间戳心跳来判断陈旧。这样消除了"重开工作区后已中断页面仍在呼吸、却没有任何恢复入口"的问题,并让【继续生成】成为所有未完成页面的统一恢复入口:保留已 accepted 页面,重新推进 interrupted、pending、基础设施失败以及内容/渲染/人工检查等失败页。

这个判定**依赖两个前提**:生成只在单实例 app 内运行(`single_instance: true`),且引擎侧只被动写入页面状态、自身永不推进。如果将来支持后台生成或多实例,必须改用心跳或运行租约(lease),否则会把真正在运行的页误标为 `interrupted`。

**考虑过的方案**

- 时间戳心跳(staleness):看 `page_progress.json` 里每页的 `updated_at`,超过阈值未更新就判为中断。拒绝,因为要引入一个魔法阈值,在慢渲染或慢 agent 场景下容易把正在跑的页误杀。
- 只在 UI 重建时把活跃态当作中断显示、不持久化。拒绝,因为重试白名单、deck 级 step 计算和续跑过滤都读持久化状态,只改 UI 会迫使这些下游逻辑各自特判活跃集合,容易不一致。
- 基于会话归属 + 持久化 `interrupted`。选择这个方案,因为生成只在单实例 app 内运行,"无主即中断"的判定无需时钟,并且把单一干净终态喂给所有下游逻辑(重试、step、续跑过滤),不必到处特判。
