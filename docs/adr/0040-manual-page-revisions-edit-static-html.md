# Manual Page Revisions edit static HTML directly

PPT App will provide manual page editing by modifying the browser-rendered static HTML DOM directly, not by converting Deck HTML into a structured presentation model, redrawing it through React, or translating edit operations back into TSX through an AST. A saved Manual Page Revision becomes that Page Generation Unit's current content and visual version, while its existing Page Source remains the authoritative authoring source that a later Agent run can revise.

Each page may own at most one current Manual Page Revision. The first implementation persists only the latest revision under `manual-edits/<page_id>/` as a manifest, the Page Source render that began the editing round, the current self-contained HTML document, an Agent-readable derivative of that document, and a validated screenshot. Reopening an existing Manual Page Revision loads its latest `current.html`; each successful save replaces the current derivatives and increments the manifest revision, while `base.html` continues to identify the Page Source render from which that manual editing round began. It does not persist cross-session undo history, operation replay, or multiple saved revisions. The manifest carries a monotonic page revision for optimistic save concurrency and records the relevant Page Source and HTML fingerprints for diagnostics.

Every save supplies the revision from which editing began. If it no longer matches the current persisted revision, the backend rejects the save without merging or publishing any submitted content. The editor retains its unsaved local state and offers only cancellation or reloading the latest persisted revision; the first release provides no force-overwrite path.

The self-contained `current.html` remains the browser, screenshot, Final Deck Render, and export input. Because its embedded binary payloads can make it unsuitable for Agent context, saving a revision also writes `current.agent.html`, which preserves the current DOM, text, and styles while replacing embedded image payloads with stable page-owned asset paths. Images added or replaced by the editor are extracted by content hash under `page-assets/<page_id>/`. The Agent derivative uses canonical absolute `file://` URLs for these assets, and the prompt supplies both the canonical absolute path and the existing Agent file-tool path mapping; a new Page Source uses the existing `LocalImage` absolute-path convention. The backend rejects asset references outside that page-owned directory, and no new asset URL protocol or copy under `template/` is introduced. Accepted Page Authoring deletes the Manual Page Revision but retains page assets referenced by the accepted Page Source, while deleting a page or starting a full Deck Generation removes the corresponding page-owned assets.

The first editor release accepts newly uploaded or replacement images only as content-verified PNG, JPEG, or WebP files, with a maximum original size of 20 MiB and maximum dimensions of 8192 by 8192 pixels. It does not accept or automatically convert GIF, SVG, or HEIC uploads. Existing page SVG and chart content is unaffected because it remains an indivisible page object. Remote image URLs remain URLs, but every referenced image must load successfully during backend save validation or the save is rejected without replacing the previous revision.

The submitted self-contained `current.html` is limited to 64 MiB after Data URL embedding, and its generated `current.agent.html` derivative is limited to 2 MiB after binary payload removal. The frontend checks the submission size before upload and the backend enforces both limits again; the first release adds no separate DOM element-count quota.

The first editor release offers only fonts whose rendering can be verified: the selected element's current font, the Deck theme font, and the fixed common set Arial, Helvetica, Times New Roman, PingFang SC, Microsoft YaHei, Noto Sans SC, and SimSun. It does not accept arbitrary font-family input or font uploads. If the managed browser cannot load the selected font during save validation, the save is rejected without replacing the previous revision.

The editor loads the complete static page document but may modify only the single fixed-size slide shell and its descendants. Workspace-local resources referenced by a Manual Page Revision are embedded into that HTML so the same file can load in the App iframe, backend Chrome, and `dom-to-pptx` without temporary local-file access or expiring resource mappings. The iframe allows same-origin DOM access for the parent App but does not allow scripts, forms, popups, navigation, or downloads; the backend cleans and validates every submitted document again before atomically publishing the HTML, screenshot, and manifest.

Manual save does not invoke the Agent Visual Review. It performs only deterministic safety and renderability validation, including sanitization, the single fixed-size slide shell, successful resource loading, and screenshot generation. A user's visually crowded or otherwise subjective layout remains valid if those deterministic checks pass; the existing Visual Review runs only after Agent-authored TSX rendering or repair.

Manual save is explicit and page-scoped. A page with unsaved changes must be saved or discarded before the editor switches pages, and a save does not wait for a whole-Deck render. After the submitted page has passed deterministic Chrome validation and its HTML and screenshot have been published, the backend atomically replaces that page's shell in the existing completed `deck.html`, preserves the completed Final Deck Render, reuses every existing page screenshot, and invalidates prior PDF/PPTX export records. Leaving the editor therefore performs no render in the normal path. If the completed Deck document is missing, structurally incompatible, or cannot be updated safely, the Manual Page Revision remains saved, Final Deck Render becomes unavailable, and leaving the editor or a later formal Review/export performs the existing full-render recovery. Restoring the Page Source version explicitly deletes that page's Manual Page Revision, rerenders only that Page Source page, and applies the same incremental Deck publication or recovery rule.

Full Final Deck Render continues to build the ordered Deck document in `ppt-engine`. It renders the current Page Sources, then replaces each manually revised page's `data-presenton-slide-shell="true"` shell with the validated shell from that page's Manual Page Revision. Incremental manual publication uses the same shell replacement contract against the already completed Deck document and is serialized per Workspace so concurrent page saves cannot overwrite one another. Review, PDF, and PPTX therefore consume the current page version through the existing `manifest.json -> deck.html -> dom-to-pptx -> deck.pptx` chain; the frontend never assembles Deck HTML or accesses Workspace files directly.

When Page Refinement or Deck Refinement targets a manually revised page, every Page Authoring attempt, including render-fix and visual-review-fix retries, receives context in this precedence order: the Manual Page Revision manifest, `current.agent.html`, `current.png`, the existing TSX, Presentation Requirements, Confirmed Outline, Workspace Style Guide, and Research Evidence or other Authoring Kit inputs. `current.agent.html` is the latest user-provided content and visual structure, `current.png` is visual context only, and the existing TSX is the older source or current repair candidate. Saved manual text and numbers are allowed user-provided content and must be preserved unless the new request explicitly changes them; neither the screenshot nor the older TSX may override them as factual grounding. The existing Visual Review remains limited to visual usability checks such as overflow, overlap, cutoff, readability, contrast, missing visuals, and canvas bounds; it does not compare the manual and newly rendered versions or judge whether manual edits were preserved. The Manual Page Revision remains current through Agent, render, and visual-review failures and is deleted only when the new Page Source render reaches the existing `accepted` state.

Editor deletion is soft only in the persisted manual document: the node remains in place with `visibility: hidden` and `data-ppt-editor-deleted="true"` so surrounding layout does not reflow. The Agent derivative preserves that node and marker as an explicit user deletion instruction, and Page Authoring must omit it from the replacement TSX rather than recovering it from the older Page Source. After a successful save the first release offers no hidden-node recovery UI; only unsaved-session undo or restoring the Page Source version can recover it.

When an edited element still participates in normal Flex, Grid, or block flow, its first move or resize promotes it to an explicitly positioned object using measured `left`, `top`, `width`, and `height` inline values. An inert, unselectable node marked `data-ppt-editor-placeholder="true"` remains in its former flow position so siblings do not reflow; elements already absolutely positioned need no placeholder. The persisted manual document and Agent derivative retain the placeholder as layout machinery, but Page Authoring must omit it from replacement TSX because it is not user content. Unsaved-session undo may restore the original flow structure.

A full Deck Generation deletes all Manual Page Revisions when it replaces the current page identities and resets Page Sources. Deleting a page deletes its Manual Page Revision, while ordinary single-page or whole-Deck refinement removes revisions only for targeted pages that reach `accepted`. No stale-revision merge workflow, HTML automatic merge, TSX AST reconciliation, or compatibility path through the removed `PptxPresentationModel` and `ppt-gener` is introduced.

Implementation selectively ports the existing structured-editor branch's full-screen shell, filmstrip, toolbar presentation, keyboard interactions, session history, save status, exit confirmation, rich-text selection utilities, styling, icons, and localized copy into a new `manual-page-editor` feature. It does not merge or cherry-pick that branch and does not port `PresentationDocument`, its React renderer, JSON document operations, presentation-document backend, or revision store; iframe DOM selection, Moveable integration, DOM operations, and Manual Page Revision persistence are implemented against the current architecture.

This decision supersedes ADR-0021 only where it described the Page Source as the sole authority for the current visible page and every static HTML snapshot as necessarily non-authoritative. ADR-0021's browser staticization, readiness, script removal, captured CSS, and managed-Chrome requirements remain active, and ADR-0034's fixed Deck shell and direct `dom-to-pptx` export contracts remain active.
