# Run generation and refinement in shadow Workspaces

Deck Generation, Page Refinement, and Deck Refinement run against a per-run shadow Workspace while the official Workspace remains the complete pre-run version. Generation Abandonment permanently fences the shadow run from becoming current and returns the user immediately to the official Workspace; best-effort Agent Session cancellation and shadow cleanup continue in the background without blocking a new run.

Each shadow Workspace is a complete, self-contained copy of its official Workspace so existing Workspace, Agent, rendering, research, manual-edit, and export contracts do not need a second partial-Workspace mode. The copy may use filesystem copy-on-write cloning when available and fall back to ordinary copying, but it must not depend on hard links or symbolic links back into the official Workspace.

Shadow preparation rebases exact official-Workspace root prefixes to the shadow root across copied text artifacts while leaving external paths, URLs, and binary content unchanged. Generation Commit performs the inverse rebase before validation and refuses publication when any shadow-root reference remains, so both the running shadow and the published official Workspace keep valid self-contained absolute references without maintaining a field-by-field path registry.

Generation content paths and diagnostic log ownership are deliberately separate. All AI, Agent, research, and storage-transfer logs continue appending to the official Workspace `.log/` throughout active, interrupted, and abandoned runs; the shadow `.log/` is only its creation-time copy. Generation Commit shares a per-official-Workspace lock with log append operations, waits for current appends, replaces the shadow log copy with the latest official `.log/`, publishes the directories, and then releases queued appends against the new official directory. This preserves existing Workspace log and Diagnostic Bundle contracts without extracting logs from abandoned shadows.

Generation Commit is owned by `ppt-engine` and uses an external transaction record rather than `page-progress.json`, because the record must remain authoritative while Workspace directories move. A run starts as `preparing` while its complete shadow copy is created and validated, becomes `active` before generation work starts, and atomically claims `active -> committing` for publication. Commit moves the official Workspace to a transaction-owned previous location, moves the validated shadow Workspace to the official path, and records `committed`; if no committed record exists after a failure or process exit, recovery restores the previous Workspace. While `committing`, Generation Abandonment is unavailable, and an ordinary commit failure is reported before the user acknowledges it and the previous Workspace is restored.

The transaction is one atomically rewritten JSON record per generation run. Its minimum durable fields are `schema_version`, stable generation `run_id`, expected `workspace_id`, `run_kind`, transaction `state`, official, shadow, and previous Workspace directory paths, optional originating `page_id`, and creation/update timestamps. `workspace_id` guards recovery and cleanup against operating on the wrong directory. Page progress, planning output, Agent Session messages or individual Agent run ids, log content, and Workspace file inventories remain in their existing owners rather than being duplicated into the transaction; a synchronous commit error is returned to the caller, while crash recovery relies only on the persisted state and directory identities.

Transaction storage is rooted at `<workspace-root>/.generation-runs/<run_id>/`, outside every official Workspace but on the same filesystem. It contains `transaction.json`, `shadow/<official-workspace-basename>/`, and, only during publication or failed-cleanup recovery, `previous/<official-workspace-basename>/`. The nested Workspace basename continues to satisfy the existing `ppt-YYYYMMDD-HHmmss` validation, while top-level Workspace discovery ignores the hidden run root and directory publication remains eligible for atomic rename.

**Considered Options**

- Edit the official Workspace in place and restore copied files on stop. Rejected because Agent and rendering work may finish after the user leaves, multi-file rollback is fragile, and the user would have to wait for safe settlement.
- Store commit recovery in `page-progress.json`. Rejected because both the official and shadow Workspace own a copy and the file itself moves during directory publication.
- Keep immutable Workspace versions behind a current-version pointer. Rejected because existing Workspace identity, absolute paths, rendering, export, and Agent file-tool conventions assume the current directory layout and would require a broader migration.

**Consequences**

- First Deck Generation saves the user's currently displayed Outline as the official Outline Draft before shadow preparation, then confirms and assigns page identities only in the shadow Workspace; abandonment, preparation failure, or later recovery therefore leaves the exact submitted draft current.
- A Deck Refinement planning result of `no_op` never enters Generation Commit: it completes as a successful no-change result, discards the shadow Workspace, and returns to the unchanged official Deck with a lightweight explanation that no optimization was needed.
- An abandoned run cannot resume, update current UI state, block a new run, or publish late results.
- Interrupted non-abandoned runs may still use their shadow Workspace for recovery.
- Transaction records and directory publication require crash-recoverable atomic file writes distinct from ordinary Workspace JSON writes.
- A successful directory publication promotes the shadow directory into the official Workspace path; afterward the previous official directory and transaction metadata are cleanup targets rather than retained Workspace history.
- After successful publication the App reopens the Workspace from its official path and replaces all shadow-path frontend snapshots; it must not continue navigating or rendering from the pre-commit shadow `WorkspaceResult`.
- Cleanup runs immediately after a committed publication but is not part of commit success. A deletion failure emits a warning and leaves the minimum committed transaction marker needed for a later engine startup to recognize and retry cleanup; it does not add a separate cleanup workflow or user-visible failure state.
- Generation Abandonment revokes commit eligibility and returns the user immediately, but it does not delete the shadow directory while an Agent run may still write to it. The app requests best-effort run cancellation, continues draining the run to its terminal frame in the background, deletes the Session and shadow Workspace afterward, and leaves any abandoned residue to the next engine startup if the current process exits first.
- Once `abandoned` is durably recorded, failures from background `session.cancel(runId)`, terminal-frame draining, `session.delete()`, or shadow-directory removal are warning-only cleanup failures. They neither reverse abandonment nor block navigation or a new foreground run; unresolved residue is retried by startup cleanup. Only failure to persist `abandoned` is a user-visible stop failure.
- Host Upload objects created as transient transport during a shadow run are outside the rollback transaction. The current App Host API has no upload-deletion operation, so abandonment neither tracks nor waits for their deletion; references confined to the discarded shadow cannot become official Workspace state, and platform lifecycle cleanup owns the remote objects. A future deletion API may add best-effort background cleanup but must not delay abandonment.
- A Workspace Diagnostic Bundle keeps the official Workspace at its existing archive path and adds `generation-run/transaction.json`, the complete current foreground or interrupted shadow Workspace, and any `previous` Workspace that actually exists. It excludes unrelated and already-abandoned cleanup residues. Collection remains read-only and non-atomic; the existing entry-stability checks cover the added run tree and fail the bundle for retry if generation mutates a collected file.
- Shadow-run support does not add a Diagnostic Bundle action to the generation page. The App retains the existing Library entry and backend capability; this feature changes archive contents, not the user-facing entry points for requesting or downloading a bundle.
- Abandoned-run logs remain in the official Workspace for diagnostics and are distinguished by run and operation identities, while their generated content and user-facing resumable history are discarded.
- After abandonment, first Deck Generation returns to its pre-confirmation Outline Draft, while Page and Deck Refinement return to the pre-run Deck and preserve the user's page and view position where possible; the UI reports only that the stopped run was not retained. Same-process abandonment may preserve in-memory preview and scroll state, while the transaction record persists only an optional originating `page_id` for restart recovery; after restart the App reopens that page when it still exists and otherwise falls back to the first page.
- Official Workspace presentation content is immutable while a generation or refinement run is active; diagnostic logs are the deliberate exception. External or out-of-band content mutation during that interval is unsupported, receives no baseline comparison or merge, and may be replaced wholesale by a successful Generation Commit.
- The App workflow enforces that immutability by withholding official-Workspace editing actions while a run is active; `ppt-engine` does not add a general mutation guard across all app-facing tools. Calls that bypass the App workflow are unsupported and do not gain conflict detection or merge semantics.
- Failure to create or validate the complete shadow copy is a run-start failure rather than an interrupted run: the official Workspace remains current, the user returns to the originating Outline or Deck after acknowledging the concrete error, and any partial shadow directory is cleaned best-effort or by a later startup scan.
- The single-instance App owns at most one non-abandoned foreground generation or refinement run at a time. Abandoned runs release foreground ownership immediately and may finish cancellation or cleanup concurrently in the background without gaining navigation, UI-update, or commit authority.
- On startup, an unfinished `active` transaction is presented as the single Interrupted Generation and must be resumed or abandoned before another foreground run begins. An unfinished `preparing` transaction is a non-resumable run-start failure whose partial shadow is cleaned, `abandoned` and `committed` transactions perform cleanup without blocking the user, and an incomplete `committing` transaction restores the previous Workspace before normal use continues.
- User-confirmed stopping returns only after `ppt-engine` has atomically persisted the transaction's transition to `abandoned`; this small local control write is the only stop-path wait. Persistence failure keeps the foreground run and commit fence locally active, reports the concrete error, and offers retry instead of claiming that abandonment succeeded.
- Opening the stop-confirmation dialog does not pause or reserve the run. The confirmation action attempts the atomic `preparing` or `active` transition to `abandoned`; if publication has already claimed `committing`, commit wins, the dialog closes, and the App presents the completed Deck instead of rolling it back. Conversely, once `abandoned` is persisted, every later commit attempt is fenced out.
- Active first-generation stopping uses the destructive confirmation title “停止生成？” and explains that every result from this run will be discarded before returning to the pre-run Outline Draft. Page and Deck Refinement use “停止优化？” and explain that all optimization results will be discarded before restoring the pre-run Deck. The secondary action is “继续生成” or “继续优化”, and the destructive action is consistently “停止并放弃”; user-facing copy does not reuse the removed cancellation terminology.
- The startup recovery choice for an Interrupted Generation is itself the destructive confirmation surface: it offers only resume and “停止并放弃”, explains that abandonment is irreversible and returns to the pre-run version, and performs the atomic abandonment transition directly without opening a third confirmation dialog.
