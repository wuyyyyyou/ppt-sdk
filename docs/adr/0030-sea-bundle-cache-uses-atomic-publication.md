# SEA bundle 缓存采用独立暂存与原子发布

`ppt-engine` SEA 进程继续按 bundle ID 跨进程共享解包缓存，避免每个进程重复解包数千个应用文件；并发首次启动时，每个进程必须在独立 staging 目录中完成解包和校验，只有完整目录才能原子发布为正式缓存。发布竞争失败的进程复用已验证的正式缓存并清理自己的 staging，进程崩溃不得删除或修改其他进程正在构建或已经发布的缓存；选择这一方案是在启动性能与并发实现复杂度之间的权衡，不使用进程私有全量缓存，也不使用覆盖整个解包过程的全局锁。

staging 发布前必须按 SEA asset manifest 对所有文件做 SHA-256 校验；正式缓存的正常启动路径只快速验证 bundle ID ready marker、SEA asset manifest、插件入口和 package manifest，以保留缓存启动收益。如果快速验证失败，应隔离正式缓存并重建；如果快速验证通过但入口加载因缓存内文件缺失或损坏失败，可以隔离、重建并自动重试一次，不能为普通插件业务异常重建缓存，也不能无限重试。

损坏的正式缓存必须先原子重命名为唯一的 `.corrupt-*` 目录，不能在共享路径上直接递归删除；成功重建后默认清理该目录，后续启动可以清理超过 24 小时的遗留 `.corrupt-*` 和 `.staging-*`。清理失败只产生诊断信息而不阻止启动，并提供显式诊断开关保留失败缓存现场；任何清理流程都不能删除可能仍被其他进程使用的有效正式缓存。

缓存磁盘格式拥有独立于 tool 版本的 schema version（格式版本）；本次原子发布协议使用新的 `presenton-template-engine-sea-v2` 临时目录命名空间，不能读取、修复或删除旧 v1 缓存。旧版与新版进程可以并行使用各自命名空间，v1 遗留交给操作系统临时目录清理；未来只有缓存格式或并发协议不兼容时才升级 schema version，不随每个 engine 版本变化。
