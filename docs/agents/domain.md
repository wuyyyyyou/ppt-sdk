# Domain Docs（领域文档）

这是工程类技能在探索代码库时读取领域文档的方式说明。

## 探索前先读这些

- 根目录的 **`CONTEXT.md`**。
- **`docs/adr/`**：读取和你要处理的区域相关的 ADR。

如果这些文件不存在，**静默继续**即可。不要提示缺失，也不要提前建议创建。`/grill-with-docs` 会在术语或决策真正落地时再懒创建它们。

## 文件结构

单上下文仓库（大多数仓库都是这种）：

```
/
├── CONTEXT.md
├── docs/adr/
│   ├── 0001-event-sourced-orders.md
│   └── 0002-postgres-for-write-model.md
└── src/
```

## 使用词汇表里的说法

当你的输出要写到某个领域概念时（比如 issue 标题、重构提案、假设、测试名），优先使用 `CONTEXT.md` 里定义的术语，不要换成词汇表明确避免的同义词。

如果你需要的概念还没进词汇表，这通常是信号：要么你在发明项目里不用的说法（建议重想），要么确实有缺口（记下来，交给 `/grill-with-docs`）。

## 标记 ADR 冲突

如果你的输出和已有 ADR 冲突，不要悄悄覆盖，直接明确指出：

> _和 ADR-0007（event-sourced orders）冲突，但值得重新讨论，因为……_
