# PPT TSX Export Safety

这个 skill 只负责本地 TSX 页面在 HTML -> PPT 导出链路中的兼容性与稳定性规则。

它负责回答的问题是：

- 哪些 HTML / CSS / SVG 写法会在导出链路里失真
- 哪些文字、卡片、图形、装饰必须保守处理
- 哪些模块应该保留为真实文本
- 哪些图形主导模块应该截图导出
- 修改 TSX 后应该如何做导出安全检查

它不负责：

- 模板选择
- `manifest.json` 装配规则
- TSX 的基础 authoring 语法
- 工具参数真值表
- HTML / model / PPTX 整体工作流

如需这些内容，分别加载：

- `ppt-workflow`
- `ppt-manifest-rules`
- `ppt-template-selection`
- `ppt-tool-reference`
- `ppt-render-export`
- `ppt-tsx-rules`

## 一、强制使用规则

只要任务需要修改或新增任何 TSX 文件，就必须阅读本 skill，并按本 skill 的要求修改。

这是强约束，不是建议。

结论：

1. 修改 TSX 时，`ppt-tsx-rules` 和 `ppt-tsx-export-safety` 必须同时生效
2. 只满足 authoring 规则，不代表导出链路安全

## 二、总原则

### 1. 把 TSX 当成结构化 slide，不是普通网页

必须满足：

1. 固定单页画布
2. 稳定布局
3. 无需交互即可完整展示
4. 首屏即完整渲染

### 2. 优先服务导出链路，而不是浏览器特效

应优先使用：

1. 简单文本
2. 简单图片
3. 简单卡片
4. 简单表格
5. 简单 SVG 装饰
6. 清晰分区布局

应避免：

1. 复杂 CSS 滤镜
2. 复杂 blend mode
3. 极端 z-index 叠加
4. 依赖浏览器高级渲染特性的视觉效果
5. 结构极度复杂的嵌套

### 3. 关键文本优先保留为真实文本

以下内容应尽量保留为真实文本节点：

1. 标题
2. 正文
3. bullet
4. 指标说明
5. 关键标签
6. 关键数字

严禁：

1. 把文本绘制到 canvas 上
2. 用整张图片承载大段正文
3. 用复杂 SVG 把大段可编辑文本变成路径图形

## 三、高风险 HTML / CSS / SVG 写法

以下写法在浏览器中通常正常，但在 HTML -> PPT 导出链路中容易失真，必须严格避免。

### 1. 不要把关键标题写成混合文本结构

不要写成：

1. 裸文本节点 + `<br />` + `<span>`

错误方向：

```tsx
<h1>金融行业<br /><span>战略分析报告</span></h1>
```

更稳妥的方式：

1. 拆成两个独立文本块

### 2. 不要把关键正文和 bullet 写成行内标签 + 裸文本节点混合结构

错误方向：

```tsx
<p><span className="font-bold">端到端数字化：</span>重塑客户旅程...</p>
```

也包括：

```tsx
<div><span>⚡</span>Cyberpunk / Vivid</div>
```

更稳妥的方式：

1. 前缀块与正文块拆开
2. 用 `flex` 或明确布局控制位置

### 3. 不要依赖单边 border 承载关键线条

不要依赖：

1. `border-bottom`
2. `border-top`
3. `border-left`

去表达关键视觉线条，例如：

1. 标题下划线
2. 分隔线
3. 强调竖条

更稳妥的方式：

1. 独立 `div`
2. `hr`
3. 绝对定位色块

### 4. L 形角标和边角装饰不要用空盒子 + 单边 border

不要用一个空 `div` 再配：

1. `border-top + border-left`
2. `border-right + border-bottom`

更稳妥的方式：

1. 两个独立绝对定位实心块
2. 小型静态 SVG
3. 小型静态图片

### 5. 不要依赖伪元素承载关键内容或关键装饰

不要依赖：

1. `::before`
2. `::after`

导出链路不会稳定提取伪元素。

### 6. 不要把关键文字写成透明描边字

不要把关键标题、页眉大字、背景大字、章节大字写成：

1. `color: transparent`
2. `-webkit-text-stroke`
3. 透明填充 + 外描边

这类写法在导出链路里容易退化成：

1. 黑色实心字
2. 丢失描边
3. 宽度测量错误
4. 非预期换行

### 7. 不要依赖 `text-shadow` 承载关键标题的主要偏移层

不要把关键标题主要写成：

1. 一层真实文本
2. 再叠加 `text-shadow` 充当第二层偏移色块 / 描边层

更稳妥的方式：

1. 使用两个独立文本层
2. 最上层主标题保持真实文本

### 8. 不要用普通 div/span 的 rotate 承载关键装饰或重复纹理

不要用：

1. `transform: rotate(...)`

在普通 `div` / `span` 上承载关键装饰或重复纹理。

更稳妥的方式：

1. 静态 SVG
2. 普通 `img`
3. 边界明确的小型纯装饰截图模块

### 9. 关键卡片和信息盒主背景不要依赖 gradient

对于会在 PPT 中被提取成独立 shape 的块级容器，不要把主背景写成：

1. `background: linear-gradient(...)`
2. `background-image: linear-gradient(...)`
3. 多层 `background` 叠加后承载卡片底色

典型包括：

1. 卡片
2. 标签块
3. 信息块
4. 时间线节点
5. 图表容器
6. 页脚短信息盒

更稳妥的方式：

1. `backgroundColor`
2. 半透明纯色 `rgba(...)`
3. 明确边框
4. 单独叠加装饰层

### 10. 整页压暗层和可读性 overlay 不要依赖 gradient

如果关键标题、副标题或信息条的可读性依赖：

1. `linear-gradient(...)`
2. `repeating-linear-gradient(...)`
3. 多层 gradient overlay

进入 PPT 后常见结果是：

1. 背景图保留
2. 压暗层丢失
3. 文字对比度下降

更稳妥的方式：

1. 使用 1 到 3 层独立的绝对定位半透明纯色块
2. 用明确尺寸的深色块加强关键阅读区

### 11. 不要假设 opacity / line-height / letter-spacing 会 1:1 还原

对关键文本和关键装饰，应优先使用更保守、结构更直接的表达方式。

尤其不要把关键文字的弱化效果主要建立在：

1. `rgba(..., alpha)`
2. `hsla(..., alpha)`
3. 文字 `opacity < 1`

对关键文字，优先直接使用更合适的纯色，而不是仅靠 alpha 弱化。

### 12. 不要让关键居中标题依赖 `left: 50%` + `translateX(-50%)`

这类写法在浏览器中通常正常，但在导出链路中，文本可能被重建成新的 textbox，随后出现：

1. 位置偏左或偏右
2. 单行宽度异常
3. 视觉中心偏离页面中心

更稳妥的方式：

1. 给文本节点明确宽度
2. 使用 `text-align: center`
3. 或放进明确宽度的容器并 `justify-content: center`
4. 同时保留 `whitespace-nowrap`

### 13. 简单 SVG 也不要依赖 currentColor 或外部 class

如果使用 SVG，推荐：

1. 在 SVG 元素内部直接写 `fill`
2. 在 SVG 元素内部直接写 `stroke`
3. 保持 SVG 自包含

不推荐：

1. `currentColor`
2. `fill-current`
3. `stroke-current`
4. 外部 class 控制关键 `fill` / `stroke`

## 四、关键文本的安全写法

### 1. 单行关键文本必须显式防止换行

对以下内容，应显式设置：

1. `whitespace-nowrap`
2. 足够宽度

典型包括：

1. KPI 数值
2. 百分比
3. 金额单位
4. 日期
5. 页码
6. 短标签
7. 全大写英文短标签
8. 单行关键标题

### 2. 居中语义要写在实际文本节点上

如果一段文字是视觉中心锚点，不要只让父容器负责居中。

更稳妥的方式是让真正承载文本的节点同时具备：

1. 明确宽度
2. `text-align: center`
3. `whitespace-nowrap`

推荐方向：

```tsx
<div className="flex h-[72px] w-[420px] items-center justify-center border px-[32px]">
  <div className="w-full whitespace-nowrap text-center text-[28px] font-bold">
    Centered Copy
  </div>
</div>
```

### 3. 固定高度单行文本盒要显式垂直居中

对于副标题框、标签条、状态块、徽章文案、页脚短语等固定高度单行文本，不要只靠上下 padding 看起来居中。

更稳妥的方式：

1. 文本容器有明确宽高
2. 使用 `flex` / `inline-flex`
3. 显式 `align-items: center`
4. 显式 `justify-content: center`
5. 同时加 `whitespace-nowrap`

## 五、图表、表格、图片规则

### 1. 图表

图表实现优先顺序：

1. 静态结构化图表块
2. 简单可控图形表达
3. 少量受控图表组件

#### 图形主导模块

图形主导模块默认应在外层容器加上：

```html
data-pptx-export="screenshot"
```

典型包括：

1. 折线图
2. 柱状图
3. 雷达图
4. 面积图
5. 饼图
6. 散点图
7. 组合图
8. 带坐标轴 / 图例 / 数据标签的可视化模块

原因：

1. 这类模块依赖细线条、坐标轴、SVG 文本或复杂几何关系
2. 逐元素提取稳定性不足

#### 文本主导模块

文本主导模块默认不要整块截图。

应优先拆成：

1. 标题
2. 正文
3. bullet
4. 编号
5. 分隔线
6. 卡片

典型包括：

1. 路线图卡片
2. 时间节点说明卡
3. 步骤流
4. 文字型 KPI 框架
5. 说明型矩阵

#### 混合型模块

只对纯图形区域截图，不要把大段文字一并包进截图。

#### 截图边界规则

采用截图导出时，必须满足：

1. 截图边界尽量收敛到单个图表卡片或单个纯图形模块
2. 不要无边界地包住大量正文区域
3. 图表容器必须显式设置尺寸、背景色、内边距和圆角等边界条件

### 2. 表格

表格可以作为结构化布局表达，但必须遵循：

1. 输入必须结构化，例如 `columns` + `rows`
2. 行列数量应受 schema 约束
3. 文本溢出风险应提前考虑
4. 若历史验证证明原生 `<table>` 不稳定，应改为 `div` 栅格组件

### 3. 图片

图片容器必须满足：

1. 尺寸明确
2. `object-fit` 策略明确
3. 图片缺失时布局不应彻底崩坏

## 六、何时可以用截图策略

只有在以下情况，才应优先考虑截图导出：

1. 模块是图形主导
2. 逐元素提取稳定性明显不足
3. 图形比可编辑文本更重要

如果主要信息来自文字，不要整块截图。

判断原则：

1. 文字主导模块：优先保留真实文本
2. 图形主导模块：可对纯图形区域截图
3. 混合模块：只截纯图形区域，不截大段文字

## 七、导出安全检查清单

在认为一个 TSX 页面已经足够安全之前，至少检查：

1. 关键文本是否保持为真实文本
2. 是否避免了标题 `<br /> + <span>` 混合结构
3. 是否避免了行内标签 + 裸文本节点混合结构
4. 是否没有依赖伪元素承载关键内容
5. 是否没有依赖单边 border 承载关键线条
6. 关键卡片、标签、信息盒是否使用纯色或半透明纯色主背景，而不是 gradient 主背景
7. 整页压暗层、扫描线或可读性 overlay 是否避免依赖 `linear-gradient(...)` / `repeating-linear-gradient(...)`
8. 是否避免用 `color: transparent` / `-webkit-text-stroke` 承载关键文字
9. 关键标题是否没有依赖 `text-shadow` 充当主要描边层或偏移色层
10. inline SVG 是否不依赖 `currentColor` / `fill-current` / 外部 class 才能成立
11. 关键单行文本是否显式 `whitespace-nowrap`
12. 关键居中文字是否把居中语义写在实际文本节点上
13. 图形主导图表是否加上 `data-pptx-export="screenshot"`
14. 截图边界是否收敛，没有包住大量正文
15. 图片容器尺寸与 `object-fit` 是否明确

## 八、最终记忆

修改 TSX 时，应始终按这条原则判断：

```text
关键文字尽量保留为文本
  -> 图形主导模块才考虑截图
  -> 结构越简单越稳定
  -> 不要把浏览器特效当成导出能力
```

如果一个效果在浏览器里很好看，但它依赖复杂 CSS、复杂 SVG、透明描边字、伪元素、gradient overlay 或 `text-shadow` 才成立，那么它大概率不适合直接进入这条导出链路。
